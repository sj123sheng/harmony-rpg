# Harmony RPG 第二阶段视觉升级设计

## 背景

第一阶段已经完成一个可玩的 HarmonyOS ArkTS / ArkUI Canvas RPG 原型，包含横屏页面、
虚拟摇杆移动、碰撞、NPC 和物件交互、对话推进，以及“点亮村口灵灯”的短任务流程。

第二阶段不扩展玩法主线，而是把当前偏调试占位的几何画面升级为更完整的程序化美术原型。
目标是在不引入图片素材的前提下，让玩家一眼能识别地图区域、建筑、角色、任务物件和当前
交互目标。

## 已确认方向

- 技术路线：继续使用 ArkTS / ArkUI Canvas 自绘。
- 素材策略：不引入真实图片素材，继续使用程序化图形。
- 玩法范围：保持第一阶段任务流程不变。
- 重点：地图层次、对象辨识度、交互反馈、任务状态反馈和横屏 UI 稳定性。
- 工程策略：以 `CanvasRenderer` 内部整理为主，轻量拆分样式常量和绘制层级，不引入重型素材系统。

## 非目标

第二阶段暂不包含：

- 背包、装备、角色养成。
- 战斗系统。
- 多地图切换。
- 存档系统。
- 外部地图编辑器。
- 图片素材加载和正式手绘资源管理。
- 复杂动画系统。
- C++ / XComponent 渲染替换。

## 视觉目标

第二阶段完成后，游戏截图应具备以下特征：

- 地面不再只是大块矩形，能看出草地、土路、底部边界和村落活动区域。
- 竹林、茶铺、古井、灵石、灵灯都有独立轮廓，不依赖颜色猜测对象身份。
- 玩家和 NPC 不再是纯矩形，至少包含身体、头部、朝向提示和脚下阴影。
- 当前可交互对象有统一且明显的高亮反馈。
- 灵石和灵灯能根据任务阶段呈现冷暖、亮灭或光晕变化。
- 任务提示、摇杆、交互按钮和对话框更像游戏 UI，而不是调试控件。

## 渲染分层

`CanvasRenderer` 仍作为渲染入口，读取 `GameStateSnapshot` 后按固定顺序绘制。

建议层级如下：

1. 屏幕背景层：绘制世界外侧背景，处理不同屏幕比例下的留边。
2. 地面层：绘制草地、土路、底部边界、轻量噪点或短线纹理。
3. 场景层：绘制竹林、茶铺、古井、围栏、石灯底座等静态对象。
4. 任务物件层：绘制井边灵石、村口灵灯，以及随任务阶段变化的状态。
5. 角色层：绘制 NPC 和玩家，包括脚下阴影、身体、头部、朝向提示。
6. 交互反馈层：绘制当前可交互对象的描边、光环或轻微脉冲。
7. UI 层：绘制任务提示面板、对话框、虚拟摇杆和交互按钮。

当前 ArkUI 层已经负责摇杆、交互按钮和对话框。第二阶段可以先保留这个边界，只优化样式；
Canvas 内部负责世界和任务提示绘制。

## 地图表现

地图继续使用现有 `VILLAGE_MAP` 尺寸和主要对象位置，避免大幅改动碰撞与任务验证。

地面表现：

- 顶部和左侧竹林边界使用不同深浅的绿色，避免看起来像纯碰撞调试区。
- 村路使用柔和土黄色，从左侧村口延伸到茶铺和灵灯区域。
- 草地区域增加少量低透明纹理点或短线，控制密度，避免影响角色可读性。
- 底部围栏区域增加横向木栏或石阶暗示，而不是单块色带。

场景对象：

- 茶铺由屋顶、墙体、门帘、招牌或窗格组成，颜色保持暖色但避免大面积单色。
- 古井由井口、内圈、石沿和阴影组成，强化“不可穿越”的视觉。
- 竹林用重复竹叶椭圆和竹竿线条形成边界节奏。
- 灵灯底座和灯芯分开绘制，完成任务后增加暖色光晕。
- 灵石用低饱和蓝灰色为基础，可调查时增加蓝色微光。

## 角色表现

角色仍使用程序化图形，不引入骨骼或贴图。

玩家：

- 绘制脚下半透明椭圆阴影。
- 身体使用主色块，头部使用肤色圆形。
- 根据朝向绘制一个简单方向提示，例如头部偏移、前方面部小点或衣摆方向。

NPC：

- 每个 NPC 保留现有区分色，但增加头部、身体和阴影结构。
- 守灯人颜色偏冷稳重，茶铺老板偏暖，阿梨偏亮。
- 被选中的 NPC 使用统一交互高亮，不再只用方框描边。

## 交互反馈

交互反馈应统一且不抢走主体画面。

- 当前可交互目标显示白金色描边或环形光晕。
- 任务关键物件可以根据任务阶段改变状态：
  - 灵石未可用：低亮、冷灰。
  - 灵石可调查：蓝色微光。
  - 灵石已调查：光感减弱，表示能量已转移。
  - 灵灯未点亮：暗棕或冷灰。
  - 灵灯可点亮：灯芯微暖。
  - 灵灯完成：暖色光晕和稳定灯芯。
- 多个目标重叠时仍由 `InteractionSystem` 选择最近目标，渲染层只读取 `nearbyInteractableId`。

## UI 设计

任务提示：

- 左上角任务提示面板缩小，减少调试感。
- 使用半透明浅色底、深色文字和较小字号。
- 阶段文本可以继续显示内部阶段名，但建议映射为中文文案，例如“询问守灯人”“寻找井边灵石”。

对话框：

- 保持底部显示，避免遮挡摇杆和交互按钮。
- 使用深色半透明底、轻量描边和稳定内边距。
- 说话人名称和正文建立层级，正文使用适合手机横屏阅读的行高。

交互按钮：

- 继续在右下角显示。
- 视觉上改为游戏圆形按钮，包含边框、按压反馈和清晰文字。
- 对话中按钮文案为“继续”，可交互但无对话时为“交互”。

摇杆：

- 保持左下角位置。
- 通过半透明底盘和更清晰的摇杆中心提升可见性。
- 不遮挡关键地图对象。

## 工程设计

第二阶段主要修改：

- `entry/src/main/ets/game/render/CanvasRenderer.ets`
- `entry/src/main/ets/pages/Index.ets`
- 必要时更新 `docs/qa/harmony-rpg-mvp-checklist.md`

建议在 `CanvasRenderer` 内整理以下私有方法：

- `drawGroundLayer`
- `drawSceneLayer`
- `drawQuestObjects`
- `drawActors`
- `drawInteractionHighlight`
- `drawQuestHint`
- `drawActor`
- `drawNpc`
- `drawPlayer`

可以增加少量本地常量，例如颜色、阴影、光晕半径、UI 尺寸。暂不新增公共渲染配置文件，
除非 `CanvasRenderer` 明显膨胀到难以维护。

玩法模块原则上不改：

- `GameState`
- `QuestSystem`
- `DialogueSystem`
- `InteractionSystem`
- `CollisionSystem`

如果视觉对象尺寸与碰撞区域明显不一致，可以小幅调整 `VillageMap` 中相关碰撞矩形，并同步
更新 QA 清单。

## 数据流

第二阶段不改变第一阶段数据流：

```text
触控输入
 -> InputController
 -> GameState 更新玩家、任务、对话和交互目标
 -> GameStateSnapshot
 -> CanvasRenderer 按视觉层级绘制世界
 -> ArkUI 叠加摇杆、交互按钮和对话框
```

渲染层只根据快照做表现，不反向修改玩法状态。

## 风险与边界

- Canvas 程序化图形过度复杂会导致 `CanvasRenderer` 难维护，因此第二阶段只做可读的
  基础形状组合，不追求正式美术精度。
- 纹理和光效密度需要克制，避免影响 NPC、玩家和交互目标可读性。
- UI 仍由 ArkUI 叠加，必须检查不同横屏比例下的位置，不只看当前截图尺寸。
- 如果加入阶段中文映射，需要确保找不到阶段时有兜底文案。

## 验证方式

自动验证：

- 继续运行现有逻辑测试，确认视觉修改没有破坏碰撞、交互和任务流转。

手动验证：

- 横屏打开游戏，确认第一眼能识别玩家、NPC、茶铺、古井、灵石和灵灯。
- 靠近每个 NPC 或物件时，高亮出现在正确对象上。
- 按任务阶段推进后，灵石和灵灯状态发生可见变化。
- 任务完成后，灵灯显示稳定暖色光晕。
- 摇杆、交互按钮、任务提示和对话框不互相遮挡。
- 对话文字在目标手机横屏下不溢出。

## 实施顺序建议

1. 整理 `CanvasRenderer` 绘制层级和样式常量。
2. 升级地面、竹林、茶铺、古井、围栏等静态地图表现。
3. 升级玩家和 NPC 的程序化角色绘制。
4. 升级灵石、灵灯和交互高亮。
5. 优化任务提示、交互按钮、对话框和摇杆样式。
6. 执行逻辑测试和手动横屏 QA。
